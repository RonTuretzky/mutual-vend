Under the Hood

Smart Contract Architecture

classDiagram
    class VendingMachine {
        +NUM_TRACKS uint8
        +MAX_STOCK_PER_TRACK uint256
        +voteToken VoteToken
        +treasuryDistributor ITreasuryDistributor
        +loadTrack(trackId, product, stock)
        +vendFromTrack(trackId, token, recipient)
        +withdrawRevenue(tokens, to, amounts)
    }

    class VoteToken {
        +MINTER_ROLE bytes32
        +mint(to, amount)
        +balanceOf(owner) uint256
        +delegates(owner) address
    }

    class TreasuryDistributor {
        +cycleLength uint256
        +distributionPercentageBps uint256
        +onPurchase(buyer, token, amount)
        +distribute()
        +withdrawRetainedRevenue(token, recipient)
    }

    VendingMachine --> VoteToken : deploys & mints
    VendingMachine --> TreasuryDistributor : notifies purchases
    TreasuryDistributor --> VoteToken : reads balances
    TreasuryDistributor --> VendingMachine : collects revenue

Three contracts working together: machine, tokens, and distribution

How Purchases Work

On-Chain Purchase Flow

sequenceDiagram
    participant C as Customer
    participant VM as VendingMachine
    participant VT as VoteToken
    participant TD as TreasuryDistributor

    C->>VM: vendFromTrack(track, token, self)
    VM->>VM: Transfer payment from customer
    VM->>VM: Decrement stock
    VM->>VT: mint(customer, amount)
    VT-->>C: Vote tokens issued
    VM->>TD: onPurchase(customer, token, amount)
    TD->>TD: Track eligible balance
    TD->>TD: Split: 80% distributable / 20% retained
    VM-->>C: Item dispensed

Every purchase mints vote tokens and tracks revenue for distribution
